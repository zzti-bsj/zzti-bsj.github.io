浅析：并行 并发 同步 异步

于CPU层面的并行和并发

  并发： 同时听音乐，打LOL。于单核CPU，来回切换执行任务，每一时刻只执行一个任务。
  并行： 同时听音乐，打LOL。于多核CPU，每个时刻执行多个任务。

线程层面的同步和异步

  同步： 阻塞当前线程，直到目标完成。
  异步： 不阻塞当前线程，异步执行任务。

浅析：TCP粘包

粘包

  由于TCP协议是面向字节流的协议，所以接收方在接收的时候不知道该接收多长字节流的消息，当客户端在很短的一段时间间隔发送多个消息的时候，会被粘在一块接收；所谓的粘包问题，是接收端不知道一次接收多少字节的数据造成的。
  negal优化算法：将数据量小的，时间间隔较短的数据一次性发送给接收方
  TCP为了提高传输效率，往往要收集到足够的数据后才发送一个TCP段，所以会造成粘包的现象。

不同客户端不会粘包

  由于tcp协议的粘包现象是在客户端把多个数据打包同时发送，所以不同客户端之间的数据不会粘包。（测过）

粘包的两种情况

1. 发送端一次发送多个数据

  1、发送端连续发送两次数据，中间没有时间间隔；
  2、发送端两次数据之间有一定时间间隔，则不会发生粘包；

粘包问题的解决方案

粘包问题的解决方案

  通俗的讲，粘包是因为接收端不知道接收的数据大小，所以会造成粘包。
  解决粘包问题的方法就是如何知道发送端接收的数据长度。

struct模块打包数据长度

  1. 通过python内置的struct模块把一段要发送的二进制数据的长度打包成一个特定长度的字节，发送这个字节数据
  2. 发送指定的数据
  3. 接收端接收这个字节数据，根据这个字节数据解包，得到真实数据长度
  4. 根据这个真实数据长度接收真实数据。

  发送端连续不断的发送数据，并且是小量的数据，
接收端知道发送端一次发送数据的长度之后，就会根据长度从缓冲区中准确读取每一次的数据，解决了粘包问题

git个人简单使用总结

git个人简单使用总结

廖雪峰的Git教程

  本人的这篇总结是言简意赅的概括了git常用的命令和常见场景的解决方案，具体的学习还请求参照廖雪峰的Git教程。

  本篇博客内容

  • 初始化git仓库
  • 添加、提交文件
  • 三个区的作用
  • 修改文件
  • 删除文件
  • 回滚操作

git简介

  git是一个分布式版本控制器。
  git是面向修改的。所谓的面向修改，比如修改文件的一句代码、修改了文件名、新建一个文件，这些都是修改。

工作区和版本库、暂存区

  这里要简单说一下工作区、版本库和暂存区三个区分区别。

  • 工作区：文件夹里能够看到的，文件夹仓库就是工作区
  • 暂存区(stage)：在版本库中的一个空间，称为暂存区，记录着git add的文件修改
  • 版本库(Responsitory)：.git是git的版本库。

初始化本地git仓库

  • git init

添加、提交文件

  添加和提交文件需要经过两个步骤。
  1. git add <filename>添加文件修改到暂存区中。
  2. git commit -m "提示信息"提交暂存区中的文件到版本库中。

撤销修改

  1. 撤销暂存区的修改
    git reset HEAD filename

  2. 撤销工作区的修改
    git checkout -- <filename>

回滚：回滚的是版本库中的commit的各个版本。
撤销：把在工作区的修改全部撤销。
1. 修改后还没有放入暂存区，撤销修改之后就和当前版本库中的一模一样。
2. 修改后放入暂存区后，又作了修改，这时候撤销修改就回到添加到暂存区后的状态。

总之，就是让这个文件回到最近一次git add或git commit的状态。

撤销修改的例子

回滚和撤销的例子

  • 情况1：自修改后还没有被放到暂存区，现在，撤销修改就回到和版本库一模一样的状态；
  • 情况2：已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态

情况1：

  • git checkout -- <filename>

情况2：

  • git reset HEAD <file>，把暂存区的修改撤销到与当前版本库相同（一键还原），重新放回工作区；
  • git checkout -- file，丢弃工作区的修改。

版本回退

回滚到某个命令
git reset --hard commmitid 回滚到某一个版本

回滚命令

  • git reset --hard^ 回滚到上一个版本
  • git reset --hard^^ 回滚到上上一个版本
  • git reset --hard~100 回滚到上一百个版本

删除文件

删除工作区的文件

  • rm <filename>

删除版本库中文件

  • git rm <filename>
  • git commmit -m "提交信息"

没有被添加到版本库中就被删除的工作区的文件，是无法恢复的

python 虚拟环境 windows

python虚拟环境

  虚拟环境的作用主要是避免多个项目之间对依赖包版本不同造成冲突，而新建的python开发环境。
  虚拟环境的使用，极大的方便了项目的迁移和部署。

创建虚拟环境

如果使用的是python3，则执行以下命令常见虚拟环境
python -m venv 虚拟环境名称

如果是Python较早的版本，首先

  1. 安装virtualenv
    pip install virtualenv
  2. 创建虚拟环境
    virtual 虚拟环境名称

开启虚拟环境

在已经创建好的虚拟环境的文件夹下，进入Scripts文件夹，开启虚拟环境
activate

退出虚拟环境

在已经创建好的虚拟环境的文件夹下，进入Scripts文件夹，退出虚拟环境
deactivate

第三方库导入导出安装

  通常在虚拟环境开发完成之后需要部署到服务器上，此时需要知道开发过程中使用到了哪些依赖包，所以需要把使用到的依赖包的名称导出到文件中，在服务器上从文件读取依赖包的名称，一键完成依赖包的安装。

导入命令：
pip freeze > requirements.txt

一键安装命令：
pip install -r requirements.txt
